## 1. 项目概述

### 项目定位

一款面向 Android 用户的**本地优先、零手动、自动记账工具**。核心能力依赖通知栏解析 + 本地规则引擎 + LLM 辅助学习。

### 实施策略

**风险前置，先验证最难的部分。**

Android 自动记账最大的不确定性不在于架构或 UI，而在于：
1. 通知栏能否稳定捕获支付事件。
2. Parser 能否在千变万化的通知文案中准确提取金额和商户。

因此，前两周四分之一的时间必须完成技术预研（PoC），确认核心链路可行后再投入完整工程。

## Phase 0：技术预研与 PoC（第 1–2 周）

### 目标
不做任何架构代码，只做**最小可行性验证**。确认通知监听器在目标机型上能稳定拿到支付通知原文。

### 关键任务

- [ ] **Task 0.1** 创建裸 `NotificationListenerService`，将收到的所有通知 `tickerText` + `extras` 序列化后输出到本地文件。
- [ ] **Task 0.2** 在至少 3 款主流手机上安装测试包（推荐：小米、华为/荣耀、OPPO/vivo 各一台）。
- [ ] **Task 0.3** 让测试者日常使用微信/支付宝支付，收集至少 **50 条真实支付通知样本**。
- [ ] **Task 0.4** 分析样本，归纳通知结构，输出《通知格式分析报告》。
- [ ] **Task 0.5** 验证 AccessibilityService 是否能作为 Notification 的 fallback，读取微信/支付宝账单详情页内容。

### 交付物

1. `poc/` 目录下的最小 Demo APK 源码。
2. `docs/notification_samples.md`：50+ 真实通知样本及解析结论。
3. `docs/poc_report.md`：包含通过/不通过的决策建议。

### 验收标准（Go/No-Go）

- [ ] 微信支付通知捕获率 ≥ 90%（以测试者主动记录的交易笔数为分母）。
- [ ] 至少 80% 的样本能直接提取出「金额 + 商户/商品名」。
- [ ] 如果通知捕获率 < 70%，**必须确认 AccessibilityService fallback 可行**，否则项目暂停或调整方向。

### 风险点

- 某些机型对微信通知做了折叠/加密处理，导致拿不到原始内容。
- 支付宝部分交易（转账到银行卡、信用卡还款）不发通知。

## Phase 1：核心引擎构建（第 3–5 周）

### 目标
搭建完整的数据捕获、解析、存储、展示链路，实现"能记账"。

### 关键任务

#### Module 1：工程骨架
- [ ] **Task 1.1** 按 `DESIGN.md` 的模块划分创建 Gradle 模块。
- [ ] **Task 1.2** 配置依赖注入（Koin 或 Hilt，推荐 Koin 更轻量）。
- [ ] **Task 1.3** 配置 Detekt / KtLint 代码规范。

#### Module 2：通知捕获（`feature:capture`）
- [ ] **Task 1.4** 生产级 `NotificationListenerService`，过滤只监听微信、支付宝包名。
- [ ] **Task 1.5** 将原始通知持久化到本地（作为 Snapshot，方便后续回归测试）。
- [ ] **Task 1.6** 设计通知去重机制（同一笔交易可能触发多条通知更新）。

#### Module 3：解析器（`feature:parser`）
- [ ] **Task 1.7** 基于 PoC 样本实现 `RegexParser`，支持微信、支付宝主流支付模板。
- [ ] **Task 1.8** 实现 Parser 返回的标准模型 `ParsedTransaction(amount, merchant, time, rawText, sourceApp)`。
- [ ] **Task 1.9** 将无法解析的文本标记为 `ParseFailed`，存入数据库待人工处理。

#### Module 4：数据存储（`core:database`）
- [ ] **Task 1.10** Room 实体设计：`Transaction`、`CategoryRule`、`ParseFailureLog`。
- [ ] **Task 1.11** 设计 Flow 查询：`observeTransactions()`、`observeRules()`。

#### Module 5：基础 UI（`feature:ledger`）
- [ ] **Task 1.12** 记账列表页（LazyColumn + key）。
- [ ] **Task 1.13** Reducer + StateFlow 数据流模式落地。
- [ ] **Task 1.14** 交易详情/编辑页（允许修改金额、商户、分类）。

### 交付物

1. 可独立运行的 App，完成一笔支付后能自动出现在列表中。
2. `feature:parser` 单元测试覆盖率 ≥ 80%。
3. `core:database` Schema 定稿。

### 验收标准

- [ ] 用户完成一次微信支付后，5 秒内 App 列表出现该笔交易。
- [ ] Parser 对已知样本的解析成功率 ≥ 95%。
- [ ] 列表滑动帧率稳定 60fps（Profiler 验证）。

## Phase 2：MVP 产品化（第 6–8 周）

### 目标
让产品从"能记账"升级为"用户愿意长期使用"。重点解决分类准确性、后台稳定性、合规性。

### 关键任务

#### Module 6：分类引擎（`feature:classification`）
- [ ] **Task 2.1** 构建种子规则库：手动整理 Top 300 常见商户关键词（基于 PoC 样本 + 公开数据）。
- [ ] **Task 2.2** 实现三层分类逻辑：Exact Match → Fuzzy/Contains Match → Unclassified。
- [ ] **Task 2.3** 用户修改分类后，自动生成一条新的 `CategoryRule` 并持久化。

#### Module 7：AI 学习闭环（`ai`）
- [ ] **Task 2.4** 选择 LLM 供应商（推荐：DeepSeek / 智谱 / OpenRouter，成本低且国内可用）。
- [ ] **Task 2.5** 编写 Prompt 与 JSON Schema 约束，确保输出可解析为 `CategoryRule`。
- [ ] **Task 2.6** 实现 LLM 调用调度：新商户首次出现 → 异步调用 → 解析并保存规则 → 对历史未分类交易回溯分类。
- [ ] **Task 2.7** 添加 LLM 调用配额限制（如日上限 10 次，防止费用超支或冷启动期爆调）。

#### Module 8：后台保活与用户引导
- [ ] **Task 2.8** 实现前台服务（Foreground Service）+ 常驻通知栏，提示"自动记账运行中"。
- [ ] **Task 2.9** 设计首次启动引导页，引导用户开启：
  - 通知使用权
  - 自启动 / 电池优化白名单
  - 锁后台（部分厂商需要）
- [ ] **Task 2.10** 实现保活状态检测：如果服务被杀，下次打开 App 时提醒用户重新授权。

#### Module 9：隐私合规
- [ ] **Task 2.11** 撰写《隐私政策》，核心声明：
  - 所有交易数据**仅存储在本地**。
  - LLM 调用时**仅上传商户名称**，不上传金额、时间、用户身份。
- [ ] **Task 2.12** App 内增加隐私政策入口和"数据导出/清除"功能。

### 交付物

1. 具备自动分类能力的 MVP APK。
2. 种子规则库（`assets/seed_rules.json`）。
3. 隐私政策文档（`docs/privacy_policy.md`）。
4. 用户引导流程 UI 完整实现。

### 验收标准

- [ ] 已分类交易的本地规则命中率 ≥ 85%（基于种子库 + 用户反馈）。
- [ ] 通知服务 24 小时存活率 ≥ 80%（在已引导开启白名单的测试机上）。
- [ ] LLM 调用成功率 ≥ 95%（网络正常时）。

## Phase 3：稳定与开源准备（第 9–10 周）

### 目标
打磨细节，完善测试与文档，达到可开源、可内测发布的质量标准。

### 关键任务

- [ ] **Task 3.1** 接入 GitHub Actions CI：编译、单元测试、Detekt 检查。
- [ ] **Task 3.2** 补充 Turbine Flow 测试。
- [ ] **Task 3.3** 补充 Snapshot 回归测试：将 PoC 阶段收集的 50+ 样本纳入测试集。
- [ ] **Task 3.4** 数据库迁移测试（验证 Schema 变更不会导致用户数据丢失）。
- [ ] **Task 3.5** 性能基准测试：千条交易列表冷启动时间 < 300ms。
- [ ] **Task 3.6** 编写完整 README（中英双语），包含截图、架构图、Build 指南。
- [ ] **Task 3.7** 整理开源许可证，确认使用的库均兼容 MIT/Apache-2.0。
- [ ] **Task 3.8** 招募 20 人内测群，收集一周真实使用反馈。

### 交付物

1. GitHub 仓库代码 + README + CI 配置。
2. 内测反馈报告（`docs/beta_feedback.md`）。
3. 稳定版 Release APK / AAB。

### 验收标准

- [ ] CI 全绿（编译 + 测试 + 代码规范）。
- [ ] 内测用户 NPS（净推荐值）≥ 40。
- [ ] 无 P0/P1 级崩溃（Crashlytics 或本地日志验证）。

## Phase 4：功能演进（第 11 周起）

### 目标
根据用户反馈和数据积累，逐步扩展功能边界。

### 推荐演进顺序

| 优先级 | 功能 | 说明 |
|--------|------|------|
| P1 | 通知解析鲁棒性增强 | 持续补充新机型的通知样本 |
| P2 | 分类学习优化 | 引入更智能的相似度匹配（如 Embedding 本地匹配）替代部分 LLM 调用 |
| P3 | 周消费 AI 报告 | 基于本地数据生成自然语言消费总结（每周调用 1 次 LLM） |
| P4 | 数据导出 | CSV / Excel 导出，方便用户做年度对账 |
| P5 | 云同步（CRDT） | 多端数据同步，需引入端到端加密 |
| P6 | 自动预算预测 | 基于历史数据预测下月支出 |

## 8. 关键技术决策矩阵

| 决策项 | 推荐方案 | 备选方案 | 决策理由 |
|--------|----------|----------|----------|
| 依赖注入 | **Koin** | Hilt | Koin 更轻量，适合模块化项目，编译速度更快 |
| 后台捕获 | **NotificationListenerService + ForegroundService** | AccessibilityService | 优先级：通知监听更优雅；仅在 PoC 不可行时 fallback 到无障碍服务 |
| 本地数据库 | **Room** | SQLDelight | Room 生态最成熟，与 Flow 集成最好 |
| 不可变集合 | **kotlinx.collections.immutable** | 手动 persistent list | 官方库，与 Compose 配合最佳 |
| LLM 供应商 | **DeepSeek / 智谱 GLM** | OpenAI / Claude | 成本低、中文理解好、国内可用 |
| CI / CD | **GitHub Actions** | 无 | 开源友好，免费额度足够 |

## 9. 风险分析与应对

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| 某些机型拿不到微信原始通知 | 高 | 致命 | Phase 0 必须验证；如果不可行，转向 AccessibilityService 方案 |
| 后台服务被系统杀死 | 高 | 严重 | 引导用户加白名单 + 前台服务 + 死亡检测与复活提示 |
| 微信/支付宝通知格式变化 | 中 | 中等 | 建立 Snapshot 回归测试，快速发现格式变动 |
| LLM 冷启动期成本超支 | 中 | 低 | 设置日调用上限 + 种子规则库覆盖高频商户 |
| 隐私合规被应用商店驳回 | 中 | 中等 | 提前撰写隐私政策，声明本地存储、最小化 LLM 上传数据 |
| 用户对常驻通知栏抵触 | 中 | 中等 | 通知栏文案设计亲和化（如"已帮你记账 12 笔"），并提供可关闭选项 |

## 10. 质量门禁

每个阶段合并到主分支前，必须通过以下检查：

- [ ] **编译门禁**：`./gradlew assembleDebug` 成功。
- [ ] **测试门禁**：Parser + Rule Engine + Reducer 单元测试全绿。
- [ ] **代码规范门禁**：Detekt 0 个 Blocker/Critical 问题。
- [ ] **性能门禁**：列表页重组率 < 5%（Compose Layout Inspector）。
- [ ] **样本门禁**：新增 Parser 逻辑必须附带至少 1 个真实通知样本测试用例。
